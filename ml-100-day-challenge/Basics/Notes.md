Day1 : The xFusionCorp Industries data science team needs a standardised Python environment for their new ML project. Set up a virtual environment
1 Create a Python virtual environment named ml-env under /root/code/ using python3 -m venv.
  >python -m venv ml-env
2. Activate the environment and install the following packages: numpy, pandas, scikit-learn, and matplotlib.
  > source ml-env/bin/activate
  > pip install pandas scikit-learn numpy matplotlib
3. Generate a requirements.txt file using pip freeze and save it at /root/code/requirements.txt
  >  pip freeze > requirements.txt
---
1. pyenv : (The Python Version Installer)
    pyenv lets you install and switch between dozens of different Python versions globally or per folder
   eg: system has  comes one Python version (e.g., 3.10), but Project A needs Python 3.9
   ```bash
   pyenv install 3.11.4      # Downloads and installs Python 3.11.4
   pyenv local 3.11.4        # Makes this specific folder use 3.11.4 automatically
   python -m venv my_project_env # this time env have 3.11.4 python version
   ```
2.pip-tools (The Lockfile Creator)
  Problem: pip freeze saves every single sub-dependency into your file, making it messy and hard to upgrade just one main library.  
  ip-tools lets you write a clean list of only your top-level packages (e.g., just requests) in a file called requirements.in. It then automatically calculates and locks down the exact sub-dependencies into a secure requirements.txt file.  
  a>
  ```bash
  pip install pip-tools
 ```` 
  b> pip install, you write a very clean, high-level list of the packages your project needs. Create a file named requirements.in and add just one popular library
  ```text
  # requirements.in
   requests
```
  c> Compile it into a Lockfile (pip-compile)
  ```bash
   pip-compile requirements.in
  ```
  ip-tools looks up requests, checks what sub-libraries it needs to work, and automatically outputs a perfect, fully pinned requirements.txt
  ```text
  

   # This file is autogenerated by pip-compile with Python 3.13
   # by the following command:
   #
   #    pip-compile requirements.in
   #
   certifi==2026.5.20
      # via requests
   charset-normalizer==3.4.7
      # via requests
   idna==3.18
      # via requests
   requests==2.34.2
      # via -r requirements.in
   urllib3==2.7.0
      # via requests
```

d> Sync Your Environment (pip-sync): Later, if you want to make sure your virtual environment exactly matches that list, run:
```bash
pip-sync
```

3. pipx (The App Installer):
   Problem: You want to install a command-line tool written in Python (like black for formatting or flake8 for linting) globally. If you use standard pip install, it will mess up your system-wide Python packages.
   Solution: pipx installs Python-based applications into their own isolated, hidden virtual environments, but automatically creates a shortcut so you can run the command from anywhere.
   ```bash
    pipx install black   # Installs Black in a sandbox, but lets you type 'black' anywhere
    ```

What is uv?
> uv is an extremely fast, all-in-one Python package and project manager written in Rust by Astral. It is designed to completely replace the traditional, fragmented Python toolchain—such as pip, venv, pip-tools, pyenv, and pipx—combining their functionalities into a single executable.Because it is built in Rust, it installs packages 10 to 100 times faster than standard pip

a> How uv replaces pyenv (Python Version Tool) : Instead of installing a separate tool like pyenv and configuring shell scripts, uv manages Python versions natively
```bash
uv python install 3.12
```
 b> How uv replaces pip-tools (The Locking Tool)  
   uv has the features of pip-compile and pip-sync built directly into its core commands  
   ```bash
   uv pip compile requirements.in -o requirements.txt
   uv pip sync requirements.txt
   ```
 c> How uv replaces pipx (The App Runner)  
   You do not need pipx to run standalone global tools anymore. uv handles it using temporary, cached environments.  
   ```
   uvx black my_script.py
   ```
   When you run uvx black, uv instantly downloads the black formatter into a isolated, hidden cache folder, runs your command, and keeps it ready for next time. It doesn't pollute your global system environment or require a permanent installation

   #### The Ultimate uv Project Workflow
   While uv provides drop-in replacements for old commands (like uv pip install), its true power shines when you let it manage a project from scratch. It uses a modern, unified workflow similar to Cargo (in Rust) or npm (in Node.js).  
   
    ```bash
     # 1. Initialize a clean new project folder
     uv init my_project
     cd my_project

      # 2. Add a dependency (uv automatically creates a venv and locks it instantly!)
      uv add requests
     ``` 
 
   ####   When you run uv add requests, uv instantly creates your virtual environment, modifies a pyproject.toml file, and generates a strict, cross-platform uv.lock file.

    > uv add requests, you no longer need a requirements.txt file at all.uv completely replaces the old requirement files with two modern, standardized files: pyproject.toml and uv.lock

   ####   What happens to the files?
    >  1. pyproject.toml (Your Input): This replaces requirements.in. It holds the clean, human-readable list of top-level libraries you want (e.g., dependencies = ["requests"]).
       2. uv.lock (The Machine Lockfile): This replaces requirements.txt. It is an automatically generated, strict, production-ready file that locks down the exact versions of requests and all its sub-dependencies. You never edit this file manually
